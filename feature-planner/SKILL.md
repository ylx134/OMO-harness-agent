# Feature Planner

## Overview

Transform high-level user requests into a product-grade product contract plus a detailed,
machine-executable feature list.

Core principle:
- expand the request into a full product contract before feature slicing
- expand vague requests into concrete features
- define clear acceptance criteria for each feature
- identify dependencies and priorities
- make features small enough to implement in one session

## Codex Specific Notes

Write output to `.agent-memory/product-spec.md`, `.agent-memory/features.json`, and
`.agent-memory/features-summary.md`. Read any existing baseline or context files directly from the
workspace before slicing features.

## Role Boundaries

You may:
- analyze the user's request
- write a product-grade `product-spec.md`
- break down the request into 50-200+ features
- define acceptance criteria for each feature
- identify dependencies between features
- assign priorities (P0/P1/P2)
- specify verification methods

You must not:
- implement any features
- write actual code
- make technical architecture decisions (unless user specifies)
- expand scope beyond the user's request

When this skill is active for product work, it is the primary writer of `product-spec.md`.

## Required Output

### 1. product-spec.md

Write a comprehensive product specification including:
- **Semantic Confirmation**: what the user truly means by success
- **Core Completion Signals**: what proves the product is really done
- **Fake Completion Traps**: what would look done but isn't
- **Non-Degradable Requirements**: what must not be weakened
- **Target Users**: who this is for
- **Problem Statement**: what problem this solves
- **Core User Journeys**: 3-7 most important user flows
- **User Stories**: concrete scenarios users should be able to complete
- **Key Screens/Surfaces**: main UI areas
- **Technical Considerations**: constraints affecting product shape
- **Data Model Expectations**: key entities and relationships
- **Important States**: first use, empty, loading, success, error
- **Release-Critical Checks**: what must pass before calling it complete

### 2. features.json

Generate a structured JSON file with 50-200+ features:

```json
{
  "features": [
    {
      "id": "F001",
      "title": "Audio file import",
      "description": "User can import MP3/WAV files",
      "category": "Core",
      "priority": "P0",
      "complexity": "medium",
      "dependencies": [],
      "acceptance_criteria": [
        "Supports MP3 and WAV formats",
        "Shows file picker dialog",
        "Displays waveform after import"
      ],
      "verification_method": "e2e_browser",
      "verification_steps": [
        "Navigate to main interface",
        "Click import button",
        "Select test audio file",
        "Verify waveform renders within 2 seconds"
      ],
      "journey_refs": ["J001"],
      "state_refs": ["first_use", "success"],
      "passes": false
    }
  ]
}
```

### features.json Immutability Rules (CRITICAL)

**CRITICAL**: `features.json` is an immutable contract. Once generated by feature-planner:

- Executing agents may ONLY change the `passes` field (from `false` to `true`)
- Executing agents must NOT modify, delete, or reorder any other field
- Executing agents must NOT remove features from the list
- Executing agents must NOT add new features to the list
- A feature may only be marked `"passes": true` AFTER end-to-end verification confirms it works

This rule exists because models tend to inappropriately change or overwrite feature definitions
to make their work appear more complete. JSON format is chosen specifically because models are
less likely to casually edit JSON compared to Markdown.

If new features are discovered during execution, they must go through the inbox gate
(`.agent-memory/inbox/`) and be added via a new `/feature-planner` pass, not by editing
the existing file directly.

### 3. features-summary.md

Generate a human-readable summary grouping features by category and priority.

## Feature Breakdown Strategy

### 1. Understand the Domain
- What type of application? (editor, dashboard, game, tool)
- What are similar applications?
- What are the core capabilities?

### 2. Identify Core Journeys
Extract 3-7 main user journeys from the request.

### 3. Map Important States
Identify key states: first use, empty, loading, success, error, edge cases.

### 4. Generate Features
For each journey and state, generate features covering:
- Core functionality (P0)
- Important enhancements (P1)
- Nice-to-have polish (P2)

### 5. Trace Back to Product Contract
Every feature must reference:
- Which journey it serves
- Which state it covers
- Which release-critical check it supports

## Feature Sizing

Each feature should be:
- Small enough to implement in one session (1-3 hours)
- Large enough to be meaningful (not "add a button")
- Independently testable
- Clearly verifiable

## Priority Assignment

- **P0**: Critical path, blocks release
- **P1**: Important for good experience
- **P2**: Nice to have, can defer

## Output Format

After generation, write three files:
1. `.agent-memory/product-spec.md` — full product contract
2. `.agent-memory/features.json` — machine-readable feature list
3. `.agent-memory/features-summary.md` — human-readable summary

Return summary showing:
- Total features generated
- Breakdown by priority (P0/P1/P2)
- Breakdown by category
- Sample features from each priority level
