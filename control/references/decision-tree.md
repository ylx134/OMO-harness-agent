# When to Use /control

## Quick Decision

| 场景 | 用 /control？ |
|------|--------------|
| 改 bug（1-2 文件） | 否 |
| 纯问答/解释 | 否 |
| 单行修改 | 否 |
| 代码审查 | 否 |
| 加新功能（3+ 文件） | **是** |
| 模块重构（影响范围不清） | **是** |
| 从零做产品 | **是** |
| 能力建设（如多语言） | **是** |
| 长期项目（可能中断） | **是** |

## 判断规则

**用 /control 当：**
- 需要规划（影响范围不明确）
- 需要独立验收（质量很重要）
- 需要跨会话恢复（可能中断多次）
- 需要多轮迭代（5+ 文件改动）
- 任务有多个 phase / gate，且需要持续保住全局约束

**不用 /control 当：**
- 范围明确、改动小
- 纯信息性任务
- 用户已用其他模式（如 ulw）

## Control 内置自动决策

Control 启动后会先运行 Task Router Gate：
- 任务太简单 → 委托 Sisyphus 直接执行
- 任务适合 harness → 启动完整流程

对于高上下文、长流程、多阶段的 harness 任务，`/control` 会要求 planner / executor / checker 子代理参与，不会静默退化成主线程单代理执行。

所以**不确定时用 /control 仍然是安全选择**，但复杂长任务会在缺少子代理能力时直接暴露阻塞，而不是假装进入完整流程。
